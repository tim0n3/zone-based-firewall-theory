What “zone-based” means on RouterOS

MikroTik doesn’t have a single “ZBF object” like some enterprise firewalls. Instead, you build it from:

  • interface-lists for zones (LAN / VPN / WAN)
  • dispatcher rules at the top of input and forward
  • per-zone chains where you describe allowed flows
  • default drop at the end of each chain
Why it’s worth it: when you revisit this in 6 months, you read “LAN → WAN allow” or “WAN → LAN only dstnat” instead of deciphering 70 interleaved rules.
# Mental model RAW : cheap drops + bypass rules (before conntrack where possible) MANGLE: mark connections/packets (QoS) NAT : dstnat (inbound) + srcnat/masquerade (outbound) FILTER: enforce security policy - INPUT = traffic to router itself - FORWARD = traffic through the router ZBF pattern: input top: established/related → invalid drop → dispatch by zone → default drop forward top: established/related → invalid drop → global blocks → dispatch by zone → default drop

Packet processing order (simplified)

Why this matters: if you drop traffic in RAW, it’s cheap (often before conntrack). If you want to block specific outbound protocols (like DoT on 853), doing it in FILTER forward is easy and readable.

Dispatcher pattern (human-readable intent)

# INPUT (to router) accept established,related,untracked drop invalid if in-interface-list=LAN jump zbf-in-lan if in-interface-list=VPN jump zbf-in-vpn if in-interface-list=WAN jump zbf-in-wan drop (default) # FORWARD (through router) accept established,related,untracked drop invalid apply global blocks (eg DoT/853) if in-interface-list=LAN jump zbf-fwd-lan if in-interface-list=VPN jump zbf-fwd-vpn if in-interface-list=WAN jump zbf-fwd-wan drop (default)
Tip: Put your “global rejects” (eg DoT, known-bad devices) above the zone dispatchers so they apply consistently.

How to extend this to more zones (IoT / Guest / Servers)

Add more interface-lists, then add a forward chain per zone (and optionally an input chain too). The page structure stays identical.

# Example new zones /interface list add name=IOT add name=GUEST add name=SERVERS # Put VLAN L3 interfaces into the right zone lists /interface list member add list=IOT interface=bridge.20 add list=GUEST interface=bridge.30 add list=SERVERS interface=bridge.40 # Then create per-zone chains: # zbf-fwd-iot, zbf-fwd-guest, zbf-fwd-servers # and add dispatchers near the top of FORWARD. # Example policy: # - Guest -> WAN only # - IoT -> WAN + DNS/NTP only, block IoT->LAN # - Servers -> LAN allowed, but restrict Server->WAN to updates only
